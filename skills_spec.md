# TVHDL Agent Tooling Spec — v1

## Context

The Ternary-VHDL repo currently encodes its build/simulate workflows as 100+ VS Code tasks in [.vscode/tasks.json](../../Documents/git-repos/Ternary-VHDL/.vscode/tasks.json) (954 lines, generated by `create_tb_tasks.py`). Each new testbench requires regenerating it, and the flow is locked to VS Code tasks — agents (and CLI users) have to reverse-engineer the right GHDL invocations from the JSON.

Inspired by [AdaCore/skills](https://github.com/AdaCore/skills) (which ships `alire/SKILL.md` + `references/*.md` so agents can drive `alr` correctly), we want to ship agent tooling **inside the Ternary-VHDL repo** that lets any AI coding agent perform the three core workflows from a natural-language request:

1. **Compile TVL** — analyze the six TVL library files in the required order with one command.
2. **Compile a project** — TVL → analyze RTL → analyze testbench → elaborate, with one command per project.
3. **Simulate a project** — Compile project → run with `--wave=*.ghw` → open in GTKWave, with one command.

**Two design constraints from the user:**

- **Tool-agnostic.** Must work for both Claude Code and GitHub Copilot. We achieve this with a shared canonical reference set + thin tool-specific entry files. AdaCore's repo only targets Claude — we go further.
- **No `tvhdl/` namespace folder.** Since the entire repo is "the TVHDL project", we don't nest references under a redundant `tvhdl/` directory. Reference docs live at a flat top-level location.

The tooling will be **generic / discovery-based**: it teaches the agent the GHDL command pattern (`-a` / `-e` / `-r`) and the layout conventions (`tvl_lib/workdir`, `<project>/workdir`, `-Ptvl_lib/workdir`) rather than enumerating every testbench. **VUnit (via `run.py`) is preferred when present**; raw GHDL is the fallback. **Synthesis is deferred to v2.**

## Layout (single source of truth + tool-specific entry points)

```
Ternary-VHDL/
├── AGENTS.md                           # Universal cross-tool entry point (agents.md standard)
├── CLAUDE.md                           # 1-line pointer: "See AGENTS.md"
├── agent-references/                   # Canonical reference docs (tool-neutral)
│   ├── prerequisites.md
│   ├── compile-tvl.md
│   ├── compile-project.md
│   ├── simulate-project.md
│   ├── vunit-flow.md
│   ├── ghdl-command-pattern.md
│   └── project-layout.md
├── .claude/
│   └── commands/                       # Claude Code slash commands
│       ├── compile-tvl.md              # → /compile-tvl
│       ├── compile-project.md          # → /compile-project
│       └── simulate-project.md         # → /simulate-project
└── .github/
    ├── copilot-instructions.md         # 1-line pointer: "See AGENTS.md"
    └── prompts/                        # Copilot reusable prompts (VS Code slash commands)
        ├── compile-tvl.prompt.md
        ├── compile-project.prompt.md
        └── simulate-project.prompt.md
```

### Why this layout

- **`AGENTS.md` at root** is the [agents.md](https://agents.md/) cross-tool convention — increasingly read natively by Claude Code, Codex CLI, Cursor, Aider, etc. Copilot doesn't read it natively (yet), so we mirror via `.github/copilot-instructions.md` which is a one-liner pointer to AGENTS.md (Copilot loads it; agent then follows the pointer).
- **`CLAUDE.md` at root** — Claude Code auto-loads it as project-wide instructions. One line: "See [AGENTS.md](AGENTS.md) for project workflows." Avoids duplication.
- **`agent-references/` at root, flat** — no `tvhdl/` wrapper. Both tools point into the same files. If we later add a second skill (e.g. synthesis), it gets its own top-level folder or just additional references; no premature namespacing.
- **`.claude/commands/*.md`** — Claude Code's slash-command mechanism. We are *not* using `.claude/skills/<name>/SKILL.md` because (a) skills require the `tvhdl/` folder name we're avoiding, (b) slash commands are the right primitive for "do this specific workflow", and (c) Copilot has a direct equivalent in `.github/prompts/`, so the two tools stay symmetric.
- **`.github/prompts/*.prompt.md`** — Copilot's reusable prompts. Surfaced as `/compile-tvl` etc. in VS Code Chat. Each file uses Copilot's frontmatter (`mode`, `description`).

Both tools' command files are **thin** — typically 5–15 lines that delegate to the matching `agent-references/<name>.md`. The reference docs are the source of truth.

## AGENTS.md — outline

Plain markdown, no frontmatter (agents.md spec). Sections:

1. **What this repo is** — one paragraph: TVHDL = VHDL library for balanced-ternary RTL, VHDL-2008, simulated with GHDL + GTKWave, optionally driven by VUnit.
2. **Prerequisites** — `ghdl --version` (≥5.1.1), `gtkwave --version` (≥3.3.100), optional `python -c "import vunit"`. Don't auto-install. Pointer to [agent-references/prerequisites.md](agent-references/prerequisites.md).
3. **Quick reference** — three-row table (compile TVL / compile project / simulate project) → command pattern → reference file.
4. **Decision rule: VUnit vs raw GHDL** — if project has `run.py` and `vunit` importable, use `python run.py …`; else raw GHDL. Detail in [agent-references/vunit-flow.md](agent-references/vunit-flow.md).
5. **Project layout conventions** — pointer to [agent-references/project-layout.md](agent-references/project-layout.md).
6. **Fallback** — for unmapped requests, read [agent-references/ghdl-command-pattern.md](agent-references/ghdl-command-pattern.md) and run `ghdl --help` / `ghdl <subcmd> --help`.

## agent-references/ — what each file holds

- **prerequisites.md** — Version checks, what to tell the user if missing (link to README + GHDL/GTKWave install pages). `mkdir -p tvl_lib/workdir` if absent.
- **compile-tvl.md** — Ordered list of 6 files (kleene_pkg, kleene_pkg-body, bal_logic, bal_logic-body, bal_numeric, bal_numeric-body) with the canonical command. Document ordering rule: spec before body; kleene → bal_logic → bal_numeric. One-liner shell loop with `|| exit 1`.
- **compile-project.md** — Generic discovery recipe:
  1. Run compile-tvl first (idempotent if up to date).
  2. `mkdir -p <project>/workdir`.
  3. ANA each `*.vhdl` in `<project>/rtl/`, then ELA the top entity.
  4. ANA each `*_tb.vhdl` in `<project>/testbench/`, then ELA the tb entity.
  - Handles both layouts seen in the repo: `rtl_designs/rtl/` (current "active" project) and `rtl_designs/<proj>/rtl/` (per-subproject, e.g. old_testbench).
- **simulate-project.md** — `ghdl -r <tb> --wave=<workdir>/<tb>_wave.ghw [--stop-time=<N>ns]`, then `gtkwave …`. Default stop-time matches tasks.json (100ns for `flip_flop_tb`; otherwise let the testbench self-terminate). On headless systems, emit the `.ghw` path and skip GTKWave.
- **vunit-flow.md** — `python run.py` (compile-and-run all), `python run.py "*<entity>*"` (filter), `--output-path <path>` per README. Reference [tvl_lib/run.py](../../Documents/git-repos/Ternary-VHDL/tvl_lib/run.py) and [rtl_designs/run.py](../../Documents/git-repos/Ternary-VHDL/rtl_designs/run.py); note env vars (`VUNIT_VHDL_STANDARD=2008`, `VUNIT_SIMULATOR=ghdl`).
- **ghdl-command-pattern.md** — Canonical templates from `.vscode/commands_single.txt`:
  ```
  ghdl -a --std=08 --work=<lib> --workdir=<dir> [-P<lib_dir>] <file.vhdl>
  ghdl -e --std=08 --work=<lib> --workdir=<dir> [-P<lib_dir>] <entity>
  ghdl -r <entity> --wave=<dir>/<entity>.ghw [--stop-time=<N>ns]
  gtkwave <dir>/<entity>.ghw
  ```
  Mandatory `--std=08` (TVL is VHDL-2008 only). `-Ptvl_lib/workdir` required for any unit using TVL. `.ghw` (not `.vcd`) required to preserve TVL's custom types in GTKWave.
- **project-layout.md** — Repo map, conventions, and `vhdl_ls.toml` library bindings (`TVL`, `defaultlib`, `vunit_lib`, `osvvm`). Workdirs are GHDL's per-library object cache; delete to force a clean rebuild.

## Tool-specific entry files

### `.claude/commands/compile-tvl.md` (and siblings)

Each is a thin slash-command file. Example shape:

```markdown
---
description: Compile the TVL library (six VHDL-2008 files in dependency order)
---

Read [../../agent-references/compile-tvl.md](../../agent-references/compile-tvl.md) and execute the documented sequence. After completion, list the contents of `tvl_lib/workdir/` to confirm the .o/.cf artifacts exist.
```

`/compile-project` and `/simulate-project` follow the same pattern, taking an optional `$ARGUMENTS` for the project/testbench name and forwarding into the matching reference doc.

### `.github/prompts/compile-tvl.prompt.md` (and siblings)

Copilot reusable-prompt format. Example:

```markdown
---
mode: agent
description: Compile the TVL library (six VHDL-2008 files in dependency order)
---

Read [../../agent-references/compile-tvl.md](../../agent-references/compile-tvl.md) and execute the documented sequence. After completion, list the contents of `tvl_lib/workdir/` to confirm the .o/.cf artifacts exist.
```

(Same body as the Claude version — the two files are essentially copies. Frontmatter differs: Copilot uses `mode: agent`, Claude uses `description` only.)

### `CLAUDE.md` and `.github/copilot-instructions.md`

Both contain one line pointing to AGENTS.md, e.g.:

```markdown
For project workflows (compiling TVL, compiling/simulating projects), see [AGENTS.md](AGENTS.md).
```

## Verification

End-to-end test of the v1 tooling once written. Repeat each test in **both** Claude Code and a Copilot-enabled VS Code session, confirming the same `agent-references/*.md` files are loaded and the same commands are produced:

1. **Discovery (no slash command)** — Open `Ternary-VHDL/` and ask in plain English: "compile the TVL library". Agent should load AGENTS.md (Claude) or copilot-instructions → AGENTS.md (Copilot), then `agent-references/compile-tvl.md`, then run the 6-command sequence. Verify `tvl_lib/workdir/` is populated.
2. **Slash command** — Type `/compile-tvl`. Same outcome, fewer tokens.
3. **Project compile** — `/compile-project flip_flop` (or "compile flip_flop"). Expect TVL compile (cached) + ANA flip_flop.vhdl + ELA flip_flop + ANA flip_flop_tb.vhdl + ELA flip_flop_tb. Confirm artifacts in workdir.
4. **Simulate** — `/simulate-project flip_flop_tb`. Expect compile-project + `ghdl -r flip_flop_tb --stop-time=100ns --wave=…flip_flop_tb_wave.ghw` + `gtkwave …`. On a headless box, agent emits the .ghw path and skips GTKWave.
5. **VUnit path** — In an env with `vunit_hdl` installed, ask "run all TVL tests". Agent picks `python tvl_lib/run.py` over raw GHDL.
6. **Negative case** — Ask "compile a project that doesn't exist". Agent reports missing `<proj>/rtl/` rather than silently doing nothing.
7. **Cross-tool parity** — For one testbench (e.g. `flip_flop_tb`), diff the command sequences produced by Claude vs Copilot. Should be identical modulo whitespace, and should match the corresponding chain in `.vscode/tasks.json`.

No automated tests for the tooling itself; verification is manual against the live repo.
